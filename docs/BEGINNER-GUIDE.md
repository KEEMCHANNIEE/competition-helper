## 0. 모든 프로젝트는 3가지로 이뤄진다

1. **코드** — 실제로 동작하는 내용 (`.py`, `.tsx` 파일)
2. **설정** — 코드가 쓸 값·환경 (`.env`, `.toml`, `.yaml`)
3. **배포** — 이걸 남들이 쓰게 서버에 올리는 법 (`deploy/`, `.github/`)

`deploy` 같은 폴더가 낯선 건, 보통 "코드"만 생각하는데 여긴 **배포까지** 들어있어서다.

## 1. 폴더 큰 지도 (각 폴더가 "무슨 방"인지)

| 폴더 | 뭐 하는 곳 | 비유 |
|---|---|---|
| `libs/contest_helper_core/` | 모두가 공유하는 **공통 약속**(데이터 모양·DB 연결) | 공용 **사전** |
| `services/api/` | **종업원** — 요청 받는 서버 | 홀 |
| `services/worker/` | **요리사** — AI 추천/계획 만드는 곳 | 주방 |
| `services/web/` | **화면** — 사용자가 보는 것 | 메뉴판 |
| `deploy/` | **올리는 법** — 포장·서버 배치 | 이삿짐·설치 매뉴얼 |
| `.github/` | **자동화** — 코드 올리면 자동 검사·배포 | 자동 컨베이어벨트 |
| `docs/` | **설명 문서** | 안내서 |

## 2. 서비스 폴더 안엔 보통 `src` 와 `tests` 가 있다 ⭐

`services/worker/` 안에 들어가면 `src/` 와 `tests/` 가 보인다. 이게 기본 구조다.

| 폴더 | 뭐냐 | 비유 |
|---|---|---|
| **`src`** | "source(원본)". **실제로 동작하는 진짜 코드.** 배포되는 건 이것. | 요리사의 **실제 레시피** |
| **`tests`** | 그 코드가 **제대로 되는지 자동 검사하는 코드.** 서비스엔 안 쓰임, 개발 중 확인용. | 요리 나가기 전 **간 보기** |

**왜 나누나?** `src`는 손님한테 나가는 진짜 음식, `tests`는 주방에서만 하는 간 보기. 섞으면 지저분해서 분리한다.

**테스트가 왜 중요?** 우리 프로젝트에선 AI 과제를 줄 때 **"실패하는 테스트"를 미리 넣어뒀다.** 담당자가 코드를 채워서 **그 테스트가 통과(초록불)되면 = 과제 완료.** 즉 테스트 = **정답지·채점 기준.**

```
services/worker/
├── src/worker/      ← 진짜 코드 (agent.py, llm.py ...)   = 레시피
└── tests/           ← 검사 코드 (test_agent.py ...)       = 간 보기
```

## 3. 파일 "종류"별 — 왜 필요한지

파일이 많아 보여도 종류는 몇 개뿐이다. 종류로 이해하면 된다.

| 종류 | 예시 | 왜 필요한가 |
|---|---|---|
| **실제 코드** | `main.py`, `agent.py`, `Recommend.tsx` | 진짜 동작 내용. 여기에 로직을 쓴다 |
| **묶음 표시** | `__init__.py` | "이 폴더는 파이썬 코드 묶음" 이라는 **빈 표찰** (아래) |
| **부품 목록** | `pyproject.toml`, `package.json` | "이 프로그램에 필요한 외부 도구" 적은 **장보기 목록** |
| **비밀값** | `.env`, `.env.example` | 비밀번호·키 (코드에 직접 쓰면 위험해서 분리) |
| **포장지** | `Dockerfile` | 프로그램을 **상자에 포장**하는 법 |
| **동시 실행** | `docker-compose.yml` | 여러 상자를 **내 컴퓨터에서 한 번에** 띄우기 |
| **클라우드 배포** | `helm/*.yaml`, `terraform/*.tf` | 진짜 **서버(클라우드)**에 올리는 법 |
| **자동화** | `.github/workflows/*.yml` | 코드 올리면 **자동 검사·배포** |
| **세부 설정** | `alembic.ini`, `tsconfig.json` | 특정 도구의 옵션 |

### `__init__.py` 는 왜 있나
파이썬에선 폴더 코드를 다른 데서 가져다 쓰려면, 그 폴더에 `__init__.py` 가 있어야 "여긴 코드 묶음" 이라고 인식한다.
- 대부분 **속이 비어있다.** 그냥 **"여기 코드 묶음 있음" 빈 스티커.**
- 신경 쓸 내용 없다. 있어야 해서 있는 것.

## 4. `deploy/` 자세히 — "배포"가 뭔데?

**배포 = 내 노트북에서만 돌던 프로그램을, 24시간 켜진 서버에 올려 "누구나 인터넷 접속" 되게 만드는 것.**

| 안에 든 것 | 단계 | 쉬운 말 |
|---|---|---|
| `docker/*.Dockerfile` | **포장** | 프로그램을 "어디서든 똑같이 도는 상자"로 |
| `docker-compose.yml` | **내 PC 연습** | 상자들(웹+서버+요리사+DB) 한 번에 띄움 |
| `helm/` (쿠버네티스) | **진짜 서버 배치** | 클라우드에서 상자 여러 개 관리 |
| `terraform/` | **서버 자체 생성** | "서버·DB 만들어줘"를 코드로 자동 (켤 때만, 비용 절약) |

> 순서: **포장(Docker) → 내 PC 연습(compose) → 클라우드 배치(helm) → 인프라 자동생성(terraform).**
> 이 "배포 풀 코스"를 직접 해보는 게 이 프로젝트의 **진짜 학습 목표**다(제품보다 이 경험이 핵심).

## 5. 한 장 요약

```
[코드]                    [설정]               [배포]
libs (공통 약속)          .env (비밀값)         deploy/docker (포장)
services/api (종업원)     pyproject (부품목록)   deploy/helm (클라우드)
services/worker (요리사)  *.toml *.yaml         deploy/terraform (서버생성)
services/web (화면)                             .github (자동화)

각 서비스 안 = src(진짜 코드) + tests(검사 코드)
```
- **만들 때**: `src` 안 코드(.py/.tsx)에 내용을 쓴다
- **확인할 때**: `tests` 로 잘 되는지 검사한다
- **연결할 때**: 설정 파일(`.env` 등)에 값을 넣는다
- **내놓을 때**: `deploy` 로 서버에 올린다

## 6. 자주 나오는 용어 사전 (한 줄)

| 용어 | 한 줄 뜻 |
|---|---|
| **DB (데이터베이스)** | 프로그램의 기억 저장소 (엑셀 표 같은 곳) |
| **App DB** | 우리 서비스가 직접 쓰는 장부 (회원·추천·할일) |
| **공모전 DB** | 공모전 정보 창고 (Supabase, 읽기만) |
| **API / 서버** | 요청을 받아 처리하는, 항상 켜진 프로그램 |
| **큐 (Redis)** | 할 일 쪽지를 쌓아두는 메모판 |
| **worker** | 큐에서 쪽지 꺼내 무거운 일(AI) 하는 프로그램 |
| **LLM (Gemini)** | 글 넣으면 글 나오는 AI 모델 (외부 서비스) |
| **에이전트 / 하네스** | LLM에 도구·기억·반복을 채워 실제로 일하게 만든 틀 |
| **Docker** | 프로그램을 통째로 포장한 상자 |
| **쿠버네티스(k8s)/Helm** | 상자들을 클라우드에서 띄우고 관리 |
| **Terraform** | 서버·DB를 코드로 자동 생성/삭제 |
| **CI/CD (.github)** | 코드 올리면 자동으로 검사·빌드·배포 |
| **배포(deploy)** | 프로그램을 서버에 올려 실제로 돌게 하기 |

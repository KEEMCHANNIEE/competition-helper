# 통합 설계 v2 — 대화형 공모전 코파일럿 (★ 최신 비전)

- 작성일: 2026-06-29
- 상태: **이 문서가 최신 비전의 기준점.** 기존 `PROJECT-OVERVIEW`(추천 중심)는 이 문서로 확장됨.
- 한 줄: **"대화하면서 추천받고, 공부하고, 계획까지 짜서 워크스페이스에 정리해주는 하나의 AI 에이전트."**

---

## 1. 무엇을 만드나 (쉬운 말)

민지 팀이 이 서비스로 하는 일:
```
① 대화 시작 → "나 AI 공모전 찾고 있어"
② 추천      → 에이전트가 맞는 공모전 추천
③ 선택      → "이거 할래!" 공모전 하나 고름
④ 공부      → "이 공모전 평가 기준이 뭐야?" "기획서 어떻게 써?" 물어보며 학습
⑤ 계획      → "마감까지 계획 짜줘. 우리 3명이야" → 주차별 할 일·역할 분담 생성
⑥ 정리      → 그 계획이 워크스페이스에 할 일(tasks)로 자동 저장
```
→ **추천은 입구일 뿐**, 핵심은 **대화로 준비를 끝까지 같이 하는 것.**

---

## 2. 에이전트의 3가지 능력 (하나의 에이전트, 모드만 다름)

| 능력 | 하는 일 | LLM | 필요한 도구 |
|---|---|---|---|
| **추천** | 관심사로 맞는 공모전 찾기 | ✅ | `semantic_search`, `search_competitions` |
| **공부** | 개념·방법 설명, 공모전 Q&A | ✅ (주로) | (정확성 위해) `get_competition_detail`, `web_search`🟡 |
| **계획** | 주제·일정·역할 정리 → 저장 | ✅ | `create_tasks`(워크스페이스에 할 일 저장) |

> "공부"는 대부분 LLM이 직접. 단 **특정 공모전 사실/최신 정보**는 도구로 가져와 환각 방지.

---

## 3. 어떻게 동작하나 (구조)

기존 구조(api·worker·큐)를 **재사용**하고, "대화"를 얹는다.

```
[web 채팅창]
   │  메시지 보냄
   ▼
[api] POST /chat ── 메시지 저장 ──► [App DB: messages]
   │  무거운 처리는 큐로
   ▼
[Redis 큐] → [worker = 대화형 에이전트]
                 │  이전 대화 기억 로드
                 │  필요시 도구 호출(검색/상세/할일생성)
                 │  Gemini로 답변 생성
                 ▼
         App DB: 답변 메시지 + (계획이면) tasks 저장
   │
   └─ web가 폴링으로 답변 받아 화면에 표시
```
- **채팅도 추천과 같은 흐름**(큐→worker→폴링)을 써서 구조가 일관됨.
- 워크스페이스는 **공모전 하나에 연결**(`workspace.contest_id`)되어, 그 안에 대화·계획·할 일이 모임.

## 5. 새로 만들 것 목록 (파일·테이블·화면)

### 공용 계약 `libs/contest_helper_core/` 🟩
- `models.py`에 추가: `Conversation`(대화), `Message`(메시지), `Workspace.contest_id`(공모전 연결). `Task`는 이미 있음(담당자·마감 컬럼 보강).
- `schemas.py`에 추가: `ChatMessageIn`, `ChatReplyOut`, `TaskOut`, `PlanOut`.

### api `services/api/` 🟩
- `chat/` 모듈: `POST /chat`(메시지 전송→큐), `GET /chat/{conversation_id}`(대화·답변 폴링).
- `workspaces/`에 할 일(tasks) CRUD + 공모전 연결.

### worker `services/worker/` 🟨 (AI 담당)
- `agent.py`: **대화형으로 확장** — 메시지+이전대화 받아 의도 파악(추천/공부/계획) → 도구 쓰고 답변.
- `mcp_tools/tasks.py`: `create_tasks`(계획을 워크스페이스 할 일로 저장).
- `mcp_tools/`에 `web_search`🟡(공부 보강).
- 대화 기억 로드 로직.

### web `services/web/` 🟦
- `pages/Chat.tsx`: 채팅 화면(추천·공부·계획 다 여기서).
- `pages/Plan.tsx`(또는 Workspace 확장): 생성된 할 일·역할·일정 보기.

## 7. 현실적 단계 (3주·파트타임 사이드)

한 번에 다 만들면 막힌다. **별 모양(계획 에이전트)을 향해 단계로:**

- **1단계 — 추천 동작** (지금 거 완성): agent 두뇌 채워 추천 한 방 나오게. *학습: 큐·worker·RAG·LLM.*
- **2단계 — 대화형으로 전환**: 채팅 + 추천/공부를 대화로. *학습: 대화 상태·메모리·tool-use.*
- **3단계 — 계획·기록**: 계획 생성 + `create_tasks`로 워크스페이스에 저장 + 화면. *학습: 에이전트가 실제 행동(쓰기)까지.*

> 🔴 MLOps 학습 목표(컨테이너·배포·관측)는 단계와 **병행**. 제품이 커져도 배포·운영 연습이 1차 목표인 건 그대로.

## 8. 한 장 요약

```
하나의 대화형 에이전트
   ├─ 추천 능력  (검색 도구)
   ├─ 공부 능력  (주로 LLM, 가끔 검색)
   └─ 계획 능력  (create_tasks → 워크스페이스 저장)

흐름: web 채팅 → api → 큐 → worker(에이전트) → App DB(대화·할일) → web 폴링
워크스페이스 = 공모전 하나에 연결된, 대화·계획·할 일이 모이는 팀 공간
```

추천은 입구, **대화로 추천·공부·계획을 끝까지 함께하는 것**이 이 프로젝트의 진짜 목표.

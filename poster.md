# ConMate — 포스터 콘텐츠 정리

> 발표 포스터 제작을 위한 콘텐츠 초안. 팀원 소개·데이터 소개 섹션은 별도 담당자가 작성.

---

## 1. 프로젝트 소개 및 목적

**"공모전 준비도, 대화 하나로."**

공모전 팀은 보통 이렇게 굴러갑니다 — 카톡방에서 공모전 링크 공유 → 각자 자료조사 → 누가 뭘 했는지 매번 물어봐야 아는 상태 → 마감 임박해서야 "이대로 괜찮은 거 맞아?" 걱정. **ConMate는 이 흩어진 과정을 하나의 대화 루프 안에 넣습니다.** 팀원이 챗봇에게 말을 걸면, 에이전트가 그 말의 의도를 스스로 판단해 검색·계획·기록·점검 중 필요한 일을 하고, 결과는 팀 전체가 보는 워크스페이스에 자동으로 쌓입니다.

즉 이 서비스의 본질은 "챗봇 UI"가 아니라 **"팀의 준비 과정 전체를 대화로 관측하고 개입하는 에이전트"** 입니다.

| 팀이 겪는 문제 | ConMate의 대응 |
|---|---|
| 공모전을 일일이 찾아다녀야 함 | 대화로 조건을 좁혀가며 추천 (한 번에 다 안 물어봄) |
| 계획·역할 분담이 말로만 오가고 안 남음 | 대화 → 주차별 할 일로 자동 저장·배정 |
| 팀장이 진행 상황을 취합하려면 일일이 물어봐야 함 | 실행 로그·진행률·리스크가 자동 집계, 대화로 바로 확인 |
| 늦어지는 걸 마감 직전에야 알아챔 | 가장 뒤처진 팀원을 자동 탐지해 리스크 진단 + 조정안 제시 |

## 2. 에이전트는 어떻게 동작하는가 (핵심)

메시지 하나가 들어오면 에이전트는 아래 5단계를 거칩니다. **이 루프 자체가 제품의 핵심**이라, 각 단계를 기능 카드처럼 보여주는 게 좋습니다.

### ① 발화 접수 — "무슨 말이든 일단 받는다"
사용자는 형식을 몰라도 됩니다. "이거 팀으로도 나가?", "계획 짜줘", "누가 뭐 했어?" 전부 같은 입력창.

### ② 의도 판단 — "이 말이 추천인지 계획인지 잡담인지 스스로 정한다"
- LLM에게 "13가지 의도 중 정확히 하나로 분류하라"고 지시(추천/계획/진행률/팀현황/리스크점검/주간리포트/워크스페이스 생성·수정/실행로그/주제제안/배경소개/작업조언/study)
- **LLM 호출이 실패해도 대화가 안 죽게** 규칙 기반 키워드 매칭으로 이중 폴백
- "할 일 클릭"처럼 서비스가 미리 아는 패턴은 LLM보다 먼저 확정(오분류 방지)

### ③ 도구 선택·실행 — "필요하면 직접 찾아보고 답한다"
여기가 이 프로젝트의 기술적 핵심입니다. 두 가지 방식이 공존합니다.

- **정해진 흐름형** (추천/계획/진행률 등): 의도별로 미리 짜인 절차를 실행 — 예) 계획 요청 → 확정 주제+팀원 배경+마감일을 모아 LLM에 "주차별로 짜라"고 시킨 뒤 파싱해서 저장
- **자율 도구 호출형** (그 외 자유 질문 = study): LLM에게 "비교/상세조회/재검색/웹검색/팀적합도/저장목록 조회" 6개 파이썬 함수를 **그대로** 넘겨줌(Automatic Function Calling). 언제 뭘 호출할지는 LLM이 각 함수의 설명(docstring)만 보고 스스로 결정 — "작년 수상작 뭐였어?" 같은 질문엔 예외 없이 웹 검색을 먼저 실행하도록 강제해 **모르는 걸 지어내지 못하게** 막음

### ④ 답변 생성 + 되돌릴 수 있는 흔적 남기기
- 추천 결과는 사용자에게 "1. 2. 3." 순번만 보여주고, 실제 DB 고유번호는 대화 기록 속에만 남겨둠 → 나중에 "1번으로 워크스페이스 만들어줘" 같은 후속 요청을 해석할 수 있게
- 리스크 진단처럼 "실행하면 되돌리기 부담스러운" 답변은 바로 반영하지 않고 **제안(proposal)** 으로만 저장 — 팀장이 승인 버튼을 눌러야 실제 일정에 반영됨

### ⑤ 팀에 전파
계획이 나오면 배정받은 팀원에게, 일정이 조정되면 영향받는 팀원에게 자동으로 알림이 쌓입니다. 팀장이 따로 공지할 필요가 없습니다.

## 3. 사용 기술 소개 (③번 동작을 가능하게 하는 것들)

| 기술 | 이게 없으면 안 되는 이유 |
|---|---|
| **Gemini `gemini-flash-lite-latest`** (google-genai) | ②의도 분류, 계획/리스크 문장 생성 등 거의 모든 판단·생성의 엔진. 가볍고 안정적인 alias 선택(무거운 모델은 과부하 잦음) |
| **Automatic Function Calling** | ③의 자율 도구 호출형을 가능케 하는 핵심. 새 도구를 추가해도 분기 코드를 늘릴 필요 없이 함수 하나 등록으로 끝 |
| **Gemini Embedding (`gemini-embedding-001`, 768차원) + pgvector** | 키워드가 안 맞아도 "의미"가 비슷한 공모전을 찾아주는 추천의 기반 |
| **DuckDuckGo 웹 검색 (`ddgs`)** | DB에 없는 사실(작년 수상작 등)을 확인 없이 답하지 않도록 강제하는 안전장치 |
| **Redis 큐 + FastAPI 202 패턴** | 에이전트 응답은 수 초 걸릴 수 있어, 사용자를 기다리게 하지 않고 "접수(202)+폴링" 구조로 UX 끊김 방지 |
| **App DB(Postgres) 상태기계** | 진행 중인 작업 상태를 메모리가 아니라 DB에 저장 → 서버가 여러 대여도, 재시작해도 안전 |

## 4. 구현 및 분석

### 4-1. 전체 아키텍처

```
[Web] React SPA (Chat / Workspace, 항상 동시 마운트)
   │ fetch + 세션 쿠키
   ▼
[api] FastAPI — /chat /recommend /workspaces  (LLM 직접 호출 없음, 즉시 202+job_id)
   │ Redis LPUSH
   ▼
[Redis] 큐 2개 (recommend / chat)
   │ BRPOP
   ▼
[worker] 에이전트
   main.py 큐 소비 + 상태기계(queued→running→done/failed)
   agent.py 의도 분류 → 핸들러 라우팅 (②③단계)
   competition_agent.py Automatic Function Calling 오케스트레이션
   progress_agent.py / team_fit.py 진행률·팀적합도
        │                        │
        ▼                        ▼
   App DB (Postgres+pgvector)   Gemini API / 웹검색
   (+ 공모전 DB는 읽기 전용으로 별도 조회)
```

### 4-2. 대표 기능 카드 (실제 채팅/워크스페이스 스크린샷 필요)

1. **단계적 추천** — 관심분야·참여형태·목표가 안 모이면 검색을 생략하고 먼저 되묻는다
2. **주차별 계획 자동 배정** — LLM이 짠 계획을 팀원에게 라운드로빈으로 나누고 알림
3. **도구 호출형 Q&A** — "혼자 나가도 돼?" "심사기준이 뭐야?" 같은 질문에 실제 도구를 호출해 근거를 확인한 뒤 답변
4. **리스크 진단 → 팀장 승인** — 가장 뒤처진 팀원을 자동 탐지, 조정안은 승인 후에만 반영
5. **실행 로그 → 자동 완료 처리** — "오늘 한 거 저장해줘" 한마디로 요약 기록 + 관련 할 일 자동 체크

## 5. 결론

- **차별점**: 검색 대행이 아니라, "무엇을 원하는지 스스로 판단하고 필요하면 직접 확인 후 답하는" 에이전트 루프가 서비스의 본체
- **성과**: 도구 호출 강제 규칙으로 환각 억제 / 상태를 전부 DB에 두어 서버 확장에 안전 / 되돌리기 어려운 변경은 승인 절차로 보호
- **한계**: 워크스페이스 일정·회의·인사이트 섹션은 아직 mock / 데모용 인증 우회 코드 존재
- **확장 가능성**: mock 섹션 실 데이터 연동, 실패 작업 재시도(백오프) 자동화
